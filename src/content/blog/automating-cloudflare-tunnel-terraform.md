Cloudflare Tunnel has two eras: Classic Tunnels and Named Tunnels. Classic Tunnels required downloading a binary, logging in, and managing certificate files by hand — none of which plays well with Infrastructure as Code. Named Tunnels changed that by exposing API endpoints and JSON credential files, making full Terraform automation possible.

This post walks through how to deploy a Tunnel end-to-end with Terraform: provisioning the tunnel, generating credentials, configuring ingress rules, and layering on Cloudflare Access for Zero Trust SSH.

## Why automate Tunnels

If you're already managing DNS, firewall rules, and compute in Terraform, a manually-provisioned tunnel is a gap in your IaC story. Automating it means:

- Tunnels come up as part of `terraform apply`, no separate steps
- Credentials are generated and injected into startup scripts at provision time
- The whole stack — compute, tunnel, DNS, access policies — can be torn down and rebuilt reproducibly
- CI/CD pipelines can spin up ephemeral tunnels for preview environments or bastion hosts on demand

## The core resources

The minimum Terraform config for a Named Tunnel is two resources:

```hcl
resource "random_id" "tunnel_secret" {
  byte_length = 35
}

resource "cloudflare_argo_tunnel" "example" {
  account_id = var.cloudflare_account_id
  name       = "example-tunnel"
  secret     = random_id.tunnel_secret.b64_std
}
```

`random_id` generates the 35-byte base64-encoded secret the tunnel uses to authenticate with Cloudflare's edge. From there, you wire the tunnel ID and credentials into a startup script using a `templatefile()` call — `cloudflared` installs as a system service and connects outbound on boot.

## Ingress rules

The tunnel's `config.yml` maps hostnames to backend services:

```yaml
ingress:
  - hostname: ssh.example.com
    service: ssh://localhost:22
  - hostname: app.example.com
    service: http://localhost:8080
  - service: http_status:404
```

The catch-all at the bottom is required — any request that doesn't match a rule returns a 404 rather than an error. This config gets templated and written during instance startup alongside the JSON credentials file.

## Zero Trust SSH on top

The real payoff is combining Tunnel with Cloudflare Access. Instead of exposing port 22 to the internet, you put an Access policy in front of the SSH hostname:

```hcl
resource "cloudflare_access_application" "ssh" {
  zone_id          = var.cloudflare_zone_id
  name             = "SSH Access"
  domain           = "ssh.example.com"
  session_duration = "24h"
}

resource "cloudflare_access_policy" "ssh_allow" {
  application_id = cloudflare_access_application.ssh.id
  zone_id        = var.cloudflare_zone_id
  name           = "Allow engineers"
  precedence     = 1
  decision       = "allow"

  include {
    email_domain = ["example.com"]
  }
}
```

Users authenticate in the browser first, then proxy their SSH connection through `cloudflared`. No inbound firewall rules, no exposed ports — the instance only makes outbound connections to Cloudflare's edge.

## The full picture

The infrastructure stack ends up being:

1. **Compute** — a GCP (or AWS) instance with a startup script
2. **Tunnel credentials** — generated by Terraform, injected at boot
3. **DNS** — a `CNAME` pointing the hostname to `<tunnel-id>.cfargotunnel.com`
4. **Access application + policy** — enforcing authentication before traffic reaches the tunnel

All of it comes up together with `terraform apply` and tears down cleanly with `terraform destroy`. The instance never has a public IP.
